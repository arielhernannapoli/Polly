┌────┬──────────────────────────────────────────────────────────────────────┐

│    │ Código: .AddStandardResilienceHandler()                              │
│    │ Ventaja: Menos código                                               │
│    │ Desventaja: No reutilizable                                         │

│    │ Código: for (var name in clientNames) { ... }                       │

│    │ Uso: 5-10 clientes similares                                        │

│    │ Ventaja: Control total                                              │
│    │ Uso: Lógica personalizada compleja                                 │

🔍 POLÍTICAS APLICADAS AUTOMÁTICAMENTE
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Cuando usas .AddCustomResilienceHandler(), se aplica:

┌────────────────────────────────────────────────────────────────────────────┐
│                                                                            │
│  🔄 RETRY (Reintento)                                                     │
│     • 3 intentos máximo                                                   │
│     • Espera: 1s → 2s → 4s (exponencial)                                 │
│     • Jitter: Distribución aleatoria para evitar "thundering herd"       │
│                                                                            │
│  🚧 CIRCUIT BREAKER (Corta-circuitos)                                     │
│     • Se abre si: 50% de fallos en ventana de 30 segundos               │
│     • Mínimo 3 requests para evaluar                                     │
│     • Se cierra cuando: Pasa tiempo suficiente sin errores              │
│                                                                            │
│  ⏱️  TIMEOUT (Tiempo máximo)                                              │
│     • 10 segundos por request                                            │
│     • Falla si tarda más                                                 │
│                                                                            │
└────────────────────────────────────────────────────────────────────────────┘

ANTES (Polly.Extensions.Http):

    var retryPolicy = HttpPolicyExtensions
        .HandleTransientHttpError()
        .WaitAndRetryAsync(retryCount: 3, ...);
    
    var circuitBreakerPolicy = HttpPolicyExtensions
        .HandleTransientHttpError()
        .CircuitBreakerAsync(...);
    
    builder.Services.AddHttpClient("BackendApi")
        .AddPolicyHandler(retryPolicy)
        .AddPolicyHandler(circuitBreakerPolicy);

    ❌ 15 líneas de configuración manual
    ❌ Vulnerable (Polly.Extensions.Http deprecated)
    ❌ Manual y repetitivo
    ❌ Difícil de mantener


DESPUÉS (Microsoft.Extensions.Http.Resilience):

    builder.Services.AddHttpClient("BackendApi")
        .AddCustomResilienceHandler();

    ✅ 1 línea (limpia)
    ✅ Seguro (Microsoft lo mantiene)
    ✅ Automático
    ✅ Fácil de mantener
    ✅ 93% menos código

P: ¿Puedo usar múltiples handlers?
R: Sí, puedes stackear varios:
   .AddHttpMessageHandler<LoggingHandler>()
   .AddHttpMessageHandler<AuthHandler>()
   .AddCustomResilienceHandler()

P: ¿Se aplica a todos los clientes sin nombre?
R: No, debes iterar. Ver OPCIÓN 3 en GUIA_HANDLERS_GLOBALES.md

P: ¿Cómo cambio los valores de reintento?
R: Crea un DelegatingHandler personalizado. Ver EJEMPLOS_AVANZADOS.cs

P: ¿Funciona en producción?
R: Sí, está optimizado para .NET 8 en producción.
